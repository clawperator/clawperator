# Docs Refactor Finalization Items

These items are intentionally deferred until after PR-4. They are not blockers for the current docs-refactor PR sequence, but they should be revisited before the project is considered fully closed out.

## Post-PR4 Cleanup

### 1. Revisit checked-in `llms-full.txt` artifacts

Current state:

- `sites/docs/static/llms-full.txt`
- `sites/landing/public/llms-full.txt`

Both files are generated by the docs build pipeline and are currently checked into the repo. That means ordinary docs edits also produce very large generated diffs in both locations.

Why this is deferred:

- this pattern predates PR-2 and was not introduced by the current branch
- the current build and publication flow expects these files to exist
- changing the artifact strategy is broader than the scope of this docs pass

Follow-up question:

- should these generated `llms-full.txt` files remain checked in, or should the pipeline generate/publish them without committing them to source control?

### 2. Revisit authored selector-flags fallback duplication

Current state:

- `docs/api/selectors.md` includes the marker `<!-- CODE-DERIVED: selector-flags -->`
- the page also includes an authored fallback paragraph beginning with `Even without the generated table, the authored contract is:`

That means the selector flag mapping currently appears twice:

- once in the generated marker expansion
- once in authored prose

Why this is deferred:

- both versions are currently correct
- the authored fallback was useful during this refactor while the generator output was being hardened
- removing it now is optional cleanup, not a correctness fix

Follow-up question:

- once the generator is considered stable, should the authored fallback mapping be reduced or removed to avoid long-term duplication drift?
